# Варианты повышения отклика приложения SmileRacersBot

## Что считать «откликом»

Для нашего проекта отклик = время от действия пользователя до видимого результата:

1. **Telegram-бот**: команда/кнопка → ответ в чате.
2. **Mini App API**: запрос фронта → JSON-ответ.
3. **UX-отклик**: как быстро пользователь видит промежуточное состояние (спиннер, optimistic update, toast).

Ниже — практичные варианты с приоритетами.

---

## Быстрые улучшения (1–3 дня)

### 1) Убрать блокирующий `Thread.sleep` при Telegram 429
Сейчас в `EmojiRaceBot.execute(...)` при 429 делается `Thread.sleep(retryAfterMillis)`, что блокирует поток обработки и может ухудшать общую отзывчивость под нагрузкой.

**Что сделать:**
- перейти на неблокирующий retry (через очередь/планировщик с отложенной отправкой);
- либо ограничить повтор только для не-критичных сообщений и быстро возвращать пользователю статус «сообщение поставлено в очередь».

**Ожидаемый эффект:** меньше «залипания» обработки апдейтов во время rate-limit.

### 2) Разделить «тяжёлый bootstrap» на 2 запроса
`/api/miniapp/bootstrap` одновременно подтягивает баланс, активные матчи, эмодзи, уведомления, админские usernames. Это удобно, но пиково тяжело.

**Что сделать:**
- оставить в `bootstrap` только критичный минимум для первого экрана;
- вынести `notifications`, `adminUsernames`, возможно `emojis` в отдельные lazy-endpoint’ы;
- на фронте догружать эти части после первого рендера.

**Ожидаемый эффект:** заметно быстрее First Meaningful Response в mini app.

### 3) Включить локальный in-memory cache для редко меняющихся данных
Примеры: список игроков (`playerRepository.findAll()`), leaderboard с коротким TTL, системные справочники.

**Что сделать:**
- добавить `spring-cache` + Caffeine;
- кэшировать read-методы с TTL 5–30 секунд там, где допустима небольшая «не-свежесть».

**Ожидаемый эффект:** снижение нагрузки на БД и стабильнее p95 latency.

### 4) Добавить корреляционный `requestId` и тайминги по ключевым endpoint’ам
Сейчас сложно быстро доказать, где именно теряется время.

**Что сделать:**
- MDC `requestId` для API/бота;
- логировать длительность ключевых сценариев (`bootstrap`, `join`, `vote`, `withdraw`);
- добавить метрики p50/p95/p99.

**Ожидаемый эффект:** прозрачная диагностика и приоритизация по данным, а не по ощущениям.

## Среднесрочные улучшения (3–10 дней)

### 5) Снизить количество последовательных запросов к БД в горячих сценариях
В нескольких местах используются цепочки repository-вызовов в одном запросе. Это повышает чувствительность к latency БД.

**Что сделать:**
- выделить hot-path и собрать данные через более «плоские» запросы (projection/DTO);
- где возможно — batch-fetch вместо N последовательных вызовов;
- проверить индексы по наиболее частым фильтрам/сортировкам (status, type, createdDate, ownerUserChatId).

**Ожидаемый эффект:** ниже задержка на backend и меньше хвостов p99.

### 6) Декомпозировать `MiniAppController`
Контроллер крупный и держит много обязанностей; со временем это обычно ведёт к регрессиям отклика.

**Что сделать:**
- выделить фасад/сервисы под bounded context (auth, race actions, notifications, profile);
- упростить каждую endpoint-функцию до «валидация → сервис → DTO».

**Ожидаемый эффект:** проще оптимизировать отдельные операции и безопаснее вносить изменения.

### 7) Асинхронная обработка тяжёлых операций
Некоторые действия (генерация истории/файла, массовые Telegram-рассылки, сложные расчёты) лучше делать вне request-thread.

**Что сделать:**
- добавить bounded executor + очередь задач;
- возвращать пользователю быстрый ack + статус выполнения;
- для Telegram — шаблон «принято в обработку» + последующее сообщение о результате.

**Ожидаемый эффект:** предсказуемая отзывчивость при всплесках нагрузки.

## Улучшения на фронте mini app (высокий UX-эффект)

### 8) Optimistic UI для безопасных действий
Для `vote`, переключения фильтров, пометки уведомлений — можно обновлять интерфейс сразу, а затем подтверждать сервером.

### 9) Skeleton/placeholder для экрана и карточек
Если данные грузятся >200–300 мс, лучше показать стабильный skeleton вместо «пустого» интерфейса.

### 10) Дебаунс и защита от double-submit
Для кнопок действий добавить:
- блок на повторный клик 300–700 мс;
- идемпотентный ключ на сервере для критичных операций.

**Ожидаемый эффект:** меньше дублированных запросов и «рывков» UI.

---

## Предлагаемый порядок внедрения

1. **Наблюдаемость:** метрики + requestId + тайминги.
2. **Быстрый выигрыш:** неблокирующий retry Telegram + облегчённый bootstrap.
3. **Стабилизация:** cache + индексы + hot-path SQL/DTO.
4. **Масштабирование:** очередь асинхронных задач.
5. **UX:** optimistic UI/skeleton/double-submit guard.

---

## Конкретные KPI, которые стоит зафиксировать

- `miniapp.bootstrap` p50/p95/p99 (цель: p95 < 400–600 мс).
- Время ответа на Telegram callback/message (цель: p95 < 700–900 мс, без учёта внешних rate-limit).
- Доля ошибок `429` и доля повторов.
- Процент дублей пользовательских действий (double-submit).
- Время до первого рендера ключевого экрана mini app.

---

## Мини-план на ближайший спринт

- День 1–2: instrumentation + dashboard + baseline.
- День 3–4: рефактор retry в Telegram и разгрузка bootstrap.
- День 5: cache для справочников и короткоживущих read-моделей.
- День 6–7: профилирование SQL hot-path + индексы/проекции.
